# Spell Quest 專案檢視與開發路線

最後更新：2026-07-10
產品階段：Alpha / Vertical Slice
目前主軸：Core Loop 2.0（三幕遠征、營地、研究與收服）

這份文件是 Spell Quest 的產品方向盤，用來記錄目前完成度、風險、設計決策與開發順序。每完成一輪重要功能，都應同步更新，而不是讓關鍵決策只留在開發對話裡。

## 一頁結論

Spell Quest 已經不是單純的拼字原型，而是一款具備可重複遊玩骨架的兒童英文學習遊戲。現有版本已經建立：

- 以中文題目、英文發音與字母操作為核心的拼字戰鬥。
- 五個地區、每區三幕、每幕五場遭遇、營地與 Boss 的遠征結構。
- 標準拼字、逐步洗牌、找多餘字母、限時與混合題型。
- 十層試煉塔，每五課形成一層題庫。
- 三十個怪物家族、三階段進化、研究、收服與圖鑑。
- 像素科幻美術、五種差異化場景、八首情境 BGM。
- GitHub Pages、PWA 與 Android 單檔版本的交付路線。

目前最重要的判斷是：**先把已成立的內容變成穩定、可理解、可長期學習的產品，不再優先追加大型模式。**

下一個產品里程碑建議定義為：

> Alpha 0.4：完成平板穩定化、學習核心 2.0、主線與試煉塔的難度分工，以及最小自動測試。

## 產品定位

### 核心玩家

- 主要玩家：8 歲左右、正在建立英文單字量的兒童。
- 次要使用者：希望看懂練習成果，但不想操作複雜後台的家長。
- 主要裝置：Android 橫向平板，其次是桌面瀏覽器。
- 單次理想體驗：能在短時間完成一幕，也能連續完成三幕形成一局。

### 核心承諾

1. 玩家的注意力放在怪物、題目與戰鬥，而不是工程式選單。
2. 答錯會被教會，不會因一次失誤被粗暴中斷。
3. 熟悉的字不應永遠佔據出題前排，弱項也不能被埋在最後地區。
4. 蒐集與養成要服務學習動機，不應替玩家自動回答。
5. 主線提供探索、收集與成就感；試煉塔提供精熟、速度與高壓挑戰。

## 目前玩家流程

### 主線遠征

```text
標題畫面
  -> 主選單
  -> 地區地圖
  -> 第一幕 5 場遭遇
  -> 營地結算 / 複習 / 繼續
  -> 第二幕 5 場遭遇
  -> 營地結算 / 複習 / 繼續
  -> 第三幕 5 場遭遇與 Boss
  -> 地區結算、研究與收服、下一區解鎖
```

### 模式分工

| 模式 | 核心目的 | 難度定位 | 主要獎勵 |
| --- | --- | --- | --- |
| 主線遠征 | 看見新字、探索地區、認識怪物 | 寬容，可反覆學習 | 地區進度、研究、收服、基礎養成資源 |
| 試煉塔 | 驗證不同題型下的真正熟練 | 嚴格，有時間與規則壓力 | 首通、樓層紀錄、稀有養成資源、成就 |
| 學習手冊 | 查字、發音、錯題與弱項練習 | 無懲罰 | 熟練度恢復與針對性複習 |
| 怪物圖鑑 | 蒐集、研究、進化與長期目標 | 無壓力 | 收藏完成度、夥伴選擇、成長展示 |

### 程式規模

- `game.js`：約 7,078 行、252 KB、333 個函式。
- `style.css`：約 9,867 行、235 KB。
- 目前沒有獨立的自動測試檔案。
- 現有戰鬥畫面是自製 Canvas 2D runtime，不是完整 Phaser 專案。

### 執行資產

- `assets/` 約 827 個檔案、72.68 MB。
- `art-source/` 約 129 個檔案、148.2 MB，應維持在 runtime 之外。
- 三十個怪物家族的必要狀態圖共 630 張，盤查時均存在。
- 怪物素材約 32.3 MB。
- 場景與圖層約 16.64 MB。
- 音訊約 12.24 MB。
- 13 份怪物 manifest 帶有 UTF-8 BOM；內容可解析，但應統一格式。
- `assets/fx/odd-shield-v1`、`odd-shield-v2` 存在未採用候選素材，應移出正式 runtime 或清楚標記。

### 題庫資料

- 本地題庫共 450 題，五個地區各 90 題。
- 類型：427 個單字、12 個片語、11 個句型。
- 長度：76 題為 3 字母以下、320 題為 4 至 7 字母、54 題為 8 至 12 字母。
- 目前所有資料的 `level` 都是 `1`，尚不能真正表達難度。
- 正規化後有 40 組重複英文答案；若沿用不同題目 ID，熟練度會被拆散。
- 試煉塔第 1 至 5 層各 40 題，第 6 至 10 層各 50 題。
- 主線目前以題庫等分計算地區範圍，和地區設定中的課次文字不完全一致；題庫增減後也可能位移。

## 遊戲性評估

### 已成立的優勢

1. **戰鬥輸入很直覺。** 點擊字母比鍵盤輸入更適合兒童和平板，也能直接把每個字母轉成攻擊回饋。
2. **怪物與場景有辨識度。** 三階進化和五區美術，已經提供比一般單字 App 更強的情緒記憶點。
3. **主線和試煉塔開始有分工。** 主線可承接新字與探索，塔則承接規則變化與熟練驗證。
4. **三幕結構改善長局節奏。** 營地是自然的休息、複習、獎勵和退出點，比每隻怪後彈出大型結算更連貫。
5. **答錯具有教學回饋。** 正確答案、逐字呈現、發音與怪物反擊，已經把錯誤轉成一次可記憶的事件。

### 目前最影響耐玩的缺口

1. **學習狀態仍然是累計分數，不是記憶模型。** 達到熟練後幾乎永久熟練，無法反映遺忘。
2. **歷史錯誤不會自然畢業。** 目前只要曾經答錯，錯題標記可能長期存在，會讓家長與孩子無法分辨「以前錯過」和「現在仍不會」。
3. **一幕題目在開始時就大致決定。** 玩家剛答錯的字，不能保證隔幾題再次回想；剛展現熟練的字也不會即時降低權重。
4. **重複答案拆成多份熟練度。** 同一英文答案出現在不同中文或課次時，學習紀錄會碎裂。
5. **獎勵還缺少消耗出口。** 收服與資源若只增加數字，完成圖鑑後會快速失去驅動力。
6. **主線失敗仍可能過硬。** 對兒童學習遊戲而言，主線應降低評級或獎勵，不宜頻繁終止整幕；高壓失敗應主要留給試煉塔。

## 可借鑑的成熟模式

以下只借用成功產品的結構原理，不直接複製其數值或介面：

- **Duolingo 的間隔複習：** 新字、弱字與到期複習字需要不同權重，熟練不是永久標籤。
- **Pokémon Quest 的遠征與營地：** 戰鬥是一段出發，營地負責成長、整理與下一輪準備；Spell Quest 的三幕營地很適合承接這個節奏。
- **Pokémon GO 的 Buddy 概念：** 玩家主動選擇喜愛怪物同行，透過一起完成練習累積成長，不替玩家自動解題。
- **Nintendo 兒童向難度原則：** 失敗後迅速重新參與、清楚提示下一步，將高壓挑戰放在玩家主動選擇的模式。
- **Roguelite 的短局決策：** 每幕提供小型選擇與明確結算，但避免把學習遊戲變成需要研究大量裝備數值的複雜系統。

## 學習核心 2.0 規格方向

### 單字資料模型

每題除了原始欄位，應建立可推導或明確保存的欄位：

- `canonicalWordId`：將相同英文答案聚合成同一學習單位。
- `regionId`：明確指定地區，不再完全依陣列等分推導。
- `lessonId`：保留原始課次，可穩定對應試煉塔。
- `difficultyTier`：綜合長度、片語、重複字母、標點與實際錯誤率。
- `answerModeHistory`：記錄玩家在哪些題型真正答對。

### 熟練狀態

建議由永久的「已掌握」改為生命週期：

```text
未見 -> 學習中 -> 待複習 -> 穩定
              \-> 弱項 -> 學習中
```

- 新字第一次答對後，隔 2 至 4 題安排一次延遲回想。
- 同一局成功回想後，再進入跨日複習，例如 1、3、7、14、30 天。
- 答錯會縮短下次複習間隔，但不把所有歷史成果歸零。
- 「錯題」只代表目前仍需處理；連續成功後應離開錯題清單。
- 熟練度要區分「標準拼字答對」與「不同規則下仍能答對」。

### 幕內選題

不要在整局開始時一次排死全部題目。每個遭遇開始前即時選題：

1. 優先處理本幕尚未看過的目標字。
2. 插入剛答錯但已間隔數題的字。
3. 補入到期複習字。
4. 少量放入穩定字維持信心與節奏。
5. 同一英文答案在短時間內避免以不同中文題重複轟炸。

## 優先級風險與行動

### P0：進入下一輪功能前必須處理

1. **Core Loop 2.0 基線穩定化**
   - 固定三幕、營地、研究與收服的存檔欄位。
   - 增加應用程式版本與存檔版本，讓平板問題可被辨識。
2. **小型平板版面**
   - 以 1024 x 600 為硬性基準。
   - 主選單底部與地區卡片不可被裁切或形成難以察覺的內層捲動。
   - 正式版隱藏 `LAB` 怪物測試入口。
3. **學習模型 2.0**
   - 建立 canonical word、難度、到期複習與錯題畢業機制。
   - 幕內改為即時選題。
4. **主線與試煉塔難度分工**
   - 主線錯誤優先降低評級、研究量或獎勵，不輕易中止整幕。
   - 試煉塔保留限時、失敗與重挑壓力。
5. **最小測試基礎**
   - 將隨機來源注入 seed，讓選題與掉落可以重現。
   - 為存檔遷移、幕進度、塔層切分與熟練生命週期建立測試。

### P1：建立長期循環與產品穩定度

- 加入主動夥伴：陪同、獲得經驗、提供非自動答題的小型能力。
- 將試煉塔產出的核心或素材變成進化與夥伴培育的主要消耗出口。
- 增加地區首通、三枚評章與少量清楚成就，不做大量低價值徽章。
- 選單與背景頁暫停 Canvas 動畫；必要時降至 30 FPS。
- 每幕只預載下一組怪物、場景與音效，使用 `decode()` 後再開始演出。
- 盤查大型 PNG，採用 WebP 或圖集降低解碼與記憶體壓力。
- 補 PWA 更新提示、存檔匯出 / 匯入、離線 BGM 與快取版本策略。
- 整併重複 CSS 規則，尤其是多次覆寫的戰鬥面板與響應式區塊。
- 正規化怪物 manifest 編碼，移出未採用的護盾候選素材。

### P2：前述基礎穩定後再追加

- 新地區、新怪物家族與第四種大型模式。
- 更複雜的裝備、屬性相剋或大量貨幣。
- 全面更換引擎。
- 大量一次性劇情與過場。

## 引擎與工程策略

### 是否改用 Phaser

現階段不建議全面遷移。現在的自製 Canvas 2D runtime 已經能順暢處理背景、怪物、炮擊、震動與轉場；全面更換引擎會先消耗大量時間重做已可用的內容，卻不會直接改善學習循環。

建議先做「可遷移的模組化」：

```text
learning-engine   題庫、權重、熟練與複習
campaign-state    地區、幕、營地、獎勵與解鎖
battle-state      題型、輸入、傷害與狀態機
save-manager      存檔版本、遷移、匯入匯出
audio-manager     BGM、音效、語音與混音
canvas-renderer   場景、怪物、VFX 與效能層
ui-controller     首頁、地圖、圖鑑、手冊與覆蓋層
```

只有當場景數、實體數與動畫狀態繼續倍增，且自製 renderer 成為明確瓶頸時，再評估將 `canvas-renderer` 替換成 Phaser；學習、存檔和進度模組不應因此重寫。

- [ ] 導入延遲回想與跨日間隔複習。
- [ ] 建立未見、學習中、待複習、弱項、穩定狀態。
- [ ] 錯題可畢業，熟練也會在到期後回到複習。
- [ ] 每場遭遇開始時即時選題。
- [ ] 家長摘要只顯示本次練習、目前弱項與進步，不堆技術數字。

完成定義：剛答錯的字會在合理間隔後再次出現，穩定字不會霸佔出題，重複答案共享同一份學習紀錄。

### Sprint C：進度與養成循環

目標：讓完成一局後有清楚的「下一次還想玩」理由。

- [ ] 主線寬容、試煉塔嚴格的失敗規則正式化。
- [ ] 加入一隻主動夥伴與同行經驗。
- [ ] 試煉塔資源成為進化與培育消耗出口。
- [ ] 加入地區首通、三評章與樓層最佳紀錄。
- [ ] 先做 10 至 15 個有意義的成就，不追求數量。

完成定義：蒐集完怪物後仍能透過夥伴培育、評章與塔紀錄得到長期目標，且不需要課金式每日壓力。

### Sprint D：效能與成品感

目標：讓中高階 Android 平板的演出穩定，所有介面像同一款遊戲。

- [ ] 選單與背景頁暫停或降低 Canvas 更新率。
- [ ] 建立幕級資產預載與圖片解碼流程。
- [ ] 轉換大型背景與重複素材格式，降低記憶體尖峰。
- [ ] 整併 CSS 覆寫並建立 UI token / style guide。
- [ ] 正式化護盾、洗牌、限時與終結炮擊 VFX。
- [ ] 建立 BGM、對話、戰鬥與語音的混音標準。

完成定義：1024 x 600 平板完成一整區時無明顯掉幀尖峰、音訊卡頓或錯位，所有主要頁面使用同一視覺語言。

### Sprint E：內容擴充

只有 Sprint A 至 D 的核心驗收通過後，才開始：

- 新地區與新場景。
- 新怪物家族與進化素材。
- 新答題模式。
- 劇情事件與特殊 Boss。

## 產品驗收指標

現階段不需要雲端分析，可先用有上限的本地 session log 與人工觀察：

- 一幕中位完成時間：6 至 10 分鐘。
- 兒童測試的主線幕完成率：目標至少 80%。
- 答錯後 2 至 4 題的延遲回想成功率可被記錄。
- 同一局中，單一熟字不應因權重問題反覆霸佔題目。
- 1024 x 600 不出現重要按鈕、字母或狀態被裁切。
- 標題、地圖、情報、戰鬥、營地與塔之間沒有 runtime error。
- 核心 PWA 在離線時可啟動，必要 BGM 與題庫可用。
- 本地紀錄有容量上限，不收集姓名、帳號或雲端個資。

## 已確立的設計決策

- 主線是探索、學習與收集；試煉塔是精熟與挑戰。
- 每區三幕、每幕五場，是目前主線的基準節奏。
- 營地是休息、複習、獎勵與退出點，不在每隻怪後打斷玩家。
- 怪物夥伴未來可以協助節奏與養成，但不能替玩家自動回答。
- 現階段維持自製 Canvas runtime，不進行全面引擎遷移。
- 題庫與進度預設本地優先，不依賴私人雲端連結。
- `assets/` 只放正式 runtime 素材；來源稿與候選稿不進正式載入路徑。

2. 加入版本 / 存檔版本並隱藏正式版 LAB。
3. 建立固定瀏覽器 smoke test 與資料驗證。
4. 處理 PWA 更新、離線音訊與存檔匯出。
5. 完成後再進入 Learning Core 2.0，不先新增新地區或大型模式。
